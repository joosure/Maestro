# Maestro 路线图

语言：[English](./ROADMAP.md) · [简体中文](./ROADMAP.zh-CN.md) · [更多语言](./LANGUAGES.md)

## 目标

Maestro 的目标很简单：

> **让 AI Agent 更容易、更安全、更可靠地进入真实工程团队。**

很多 Coding Agent 已经可以写代码。但团队真正需要解决的问题不只是“让 Agent 写代码”：

- 任务要来自真实项目系统，例如 TAPD、Linear 或未来更多平台；
- 代码要来自明确配置的 Git 仓库和分支；
- 每次执行要有独立工作区，避免多个任务互相干扰；
- 人要看得懂 Agent 做了什么、改了什么、为什么失败；
- 高风险步骤要能被审核；
- 团队要能逐步扩大使用范围，而不是一开始就完全放开权限。

这份路线图不按内部模块组织，而是按用户能获得的价值组织。

---

## 近期：让更多人更快跑起来

第一次接触 Maestro 的用户，应该不需要理解完整架构，也能快速看懂并跑起来。

计划包括：

- 更简单的本地 demo；
- 更清楚的 Quick Start；
- 增加截图、GIF 或短视频演示；
- 增加能跑完整流程的示例任务；
- 解释独立工作区的价值：并行、隔离、可清理、可复盘；
- 解释为什么部分命令仍然使用 `symphony` 兼容命名；
- 提供从本地 demo 过渡到真实项目配置的清晰路径。

希望更容易演示这些场景：

- TAPD 任务到 GitHub PR；
- Linear 任务到 GitHub PR；
- 写代码前先分析需求；
- 自动分诊任务；
- 推荐 reviewer；
- 对比 Codex、Claude Code 和 OpenCode 在相似任务上的表现。

成功标准是：新读者几分钟内就能回答：

> “Maestro 是做什么的？为什么我的团队可能需要它？”

---

## 下一阶段：让 Agent 接入真实项目流程

Maestro 应该帮助 Agent 从团队已经在用的项目系统开始工作，而不是要求团队重新发明一个任务队列。

计划包括：

- 改善现有 TAPD 和 Linear 流程；
- 让任务状态、评论、链接和结果更容易理解；
- 让流程模板更容易发现、复制和修改；
- 支持更多常见研发任务，例如 Bug、小功能、需求分析、需求澄清、任务分诊和评审建议；
- 更清楚地区分“当前已有接入能力”和“未来可扩展方向”；
- 为更多集成做准备，例如 Jira、GitHub Issues、GitLab、Gitea、Bitbucket 和飞书项目。

成功标准是：团队可以从已有项目流程开始使用 Agent，而不是为了 Agent 改变团队的项目管理方式。

---

## 中期：让 Agent 执行过程更可信

团队不能只因为 Agent 说“完成了”就相信结果。

计划包括：

- 更清楚的运行历史；
- 更容易阅读的执行摘要；
- 更好地关联任务、Git 变更、日志和评审材料；
- 更清楚的失败原因；
- 更好的日志脱敏；
- 更实用的 dashboard；
- 在写入真实项目系统、推送分支或创建 PR 前提供可见的检查点；
- 更清楚地区分本地 demo、可信评估、团队试点和生产运行。

成功标准是：reviewer 能回答：

- Agent 做了什么？
- 它基于哪个任务和哪个 Git 仓库工作？
- 它改了什么？
- 它为什么停下来？
- 还有什么需要人工确认？
- 现在继续是否安全？

---

## 长期：让团队可以规模化使用 Agent

单个 Agent demo 很有价值，但团队级使用需要更强的运营能力。

计划包括：

- 安全地同时运行多个任务；
- 让不同项目、不同任务拥有隔离的工作区和执行记录；
- 根据任务类型选择不同 Agent；
- 更清楚地管理账号、凭据、额度和成本；
- 改善团队级执行环境；
- 提供更好的重试和恢复能力；
- 支持更清楚的人工审批关卡；
- 帮团队比较不同 Agent 和不同流程的实际效果。

成功标准是：团队可以逐步扩大 Agent 使用范围，同时仍然控制安全、成本和质量。

---

## 文档与社区

Maestro 应该先让人看懂，再让人觉得强大。

计划包括：

- 保持主 README 简短，并以例子为主；
- 把深入技术细节放到独立文档里；
- 重点维护英文和简体中文版本；
- 保留其他语言版本，并欢迎社区改进；
- 增加接入项目系统、Agent、代码平台和流程模板的贡献指南；
- 发布更多真实工程场景示例。

成功标准是：贡献者不用先读完整个代码库，也能找到合适的入口。

---

## 暂不追求什么

Maestro 不会把第一目标设定为“替团队跳过评审、测试和发布判断”。

我们更关注：

- 让 Agent 接入真实任务；
- 让代码来源和任务来源清楚可见；
- 让执行过程可追踪；
- 让人在高风险步骤上保持控制；
- 保留有用的执行记录；
- 随着证据逐步扩大自动化范围。

自动化应该随着可信度增长，而不是靠想象直接放大。

---

## 当前重点

当前重点是让 Maestro 更容易理解、更容易试用、更安全评估：

1. 简化公开 README；
2. 增加通俗版路线图；
3. 改善本地 demo 指引；
4. 说明当前已有接入能力，而不是把外部系统说成“内置”；
5. 解释独立工作区的意义；
6. 增加 TAPD、Linear、GitHub、CNB 等真实组合示例；
7. 保留技术细节，但不要让每个新读者一上来就面对它们。

---

## 如何贡献

有价值的贡献包括：

- 更好的示例；
- 更清楚的文档；
- 更安全的流程模板；
- 新的项目系统接入；
- 新的 Coding Agent 接入；
- 新的代码平台接入；
- dashboard 改进；
- 真实工作流测试覆盖；
- 母语者参与翻译校对。

建议先从本地 memory/mock 流程开始，再逐步接入真实系统。
