# Maestro Roadmap

언어: [English](./ROADMAP.md) · [简体中文](./ROADMAP.zh-CN.md) · [한국어](./ROADMAP.ko.md) · [More](./LANGUAGES.md)

## 목표

Maestro의 목표는 단순합니다.

> **AI Agent를 실제 엔지니어링 팀에서 더 쉽게, 더 안전하게, 더 신뢰할 수 있게 사용하도록 만든다.**

많은 Coding Agent는 이미 코드를 작성할 수 있습니다. 하지만 팀에 필요한 것은 코드 생성 그 이상입니다.

- 작업은 TAPD, Linear, 앞으로의 프로젝트 시스템 같은 실제 시스템에서 와야 합니다.
- 코드는 명시적으로 설정된 Git 저장소와 브랜치에서 와야 합니다.
- 각 실행에는 격리된 작업 공간이 있어야 합니다. 작업들이 서로 간섭하지 않기 위해서입니다.
- 사람은 Agent가 무엇을 했고, 무엇을 변경했고, 왜 실패했는지 이해할 수 있어야 합니다.
- 고위험 단계는 검토 가능해야 합니다.
- 팀은 첫날부터 모든 권한을 열지 않고 점진적으로 사용 범위를 넓힐 수 있어야 합니다.

이 로드맵은 내부 모듈 이름이 아니라 사용자 가치를 기준으로 구성되어 있습니다.

---

## 단기: Maestro를 더 쉽게 시험해 볼 수 있게 만들기

처음 접하는 사용자가 전체 아키텍처를 먼저 배울 필요 없이 Maestro를 이해하고 실행할 수 있어야 합니다.

계획:

- 더 단순한 로컬 데모.
- 더 명확한 Quick Start.
- 스크린샷, GIF, 짧은 데모 영상.
- 전체 흐름을 보여주는 예시 작업.
- 격리된 작업 공간의 가치를 명확히 설명: 병렬성, 격리, 정리, 검토 가능성.
- 남아 있는 `symphony` 호환 이름 설명.
- 로컬 데모에서 실제 프로젝트 설정으로 이어지는 명확한 경로.

더 쉽게 보여주고 싶은 시나리오:

- TAPD 작업에서 GitHub Pull Request로.
- Linear 작업에서 GitHub Pull Request로.
- 코딩 전 요구사항 분석.
- 들어오는 작업 분류.
- reviewer 제안.
- 비슷한 작업에서 Codex, Claude Code, OpenCode 비교.

성공 기준은 새 독자가 몇 분 안에 다음을 답할 수 있는 것입니다.

> “Maestro는 무엇을 하고, 왜 우리 팀에 필요할 수 있는가?”

## 다음 단계: Agent를 실제 프로젝트 워크플로에 연결

Maestro는 팀이 이미 사용하는 프로젝트 시스템에서 Agent가 일하도록 해야 합니다. Agent를 쓰기 위해 새 작업 큐를 만들라고 강요해서는 안 됩니다.

계획:

- 현재 TAPD와 Linear 흐름 개선.
- 작업 상태, 댓글, 링크, 결과를 더 이해하기 쉽게 만들기.
- workflow template을 더 찾기 쉽고, 복사하기 쉽고, 수정하기 쉽게 만들기.
- Bug 수정, 작은 기능, 요구사항 분석, 작업 정리, 분류, 리뷰 제안 같은 일반적인 엔지니어링 작업을 더 많이 지원.
- 현재 지원되는 연동과 미래 확장 대상을 명확히 구분.
- Jira, GitHub Issues, GitLab, Gitea, Bitbucket, Feishu Project 같은 더 많은 연동 준비.

성공 기준은 팀이 Agent를 쓰기 위해 작업 관리 방식을 바꾸지 않아도 되는 것입니다.

---

## 중기: Agent 작업을 더 신뢰할 수 있게 만들기

Agent가 “완료”라고 말했다는 이유만으로 팀이 실행 결과를 신뢰해서는 안 됩니다.

계획:

- 더 명확한 실행 이력.
- 읽기 쉬운 실행 요약.
- 작업, Git 변경, 로그, 리뷰 자료 사이의 더 나은 연결.
- 더 명확한 실패 원인.
- 더 나은 로그 마스킹.
- 더 유용한 dashboard.
- 실제 프로젝트 시스템에 쓰기, 브랜치 push, PR 생성 전에 보이는 체크포인트 제공.
- 로컬 데모, 신뢰된 평가, 팀 파일럿, 운영 환경을 더 명확히 구분.

성공 기준은 리뷰어가 다음을 답할 수 있는 것입니다.

- Agent가 무엇을 했는가?
- 어떤 작업과 Git 저장소에서 일했는가?
- 무엇이 변경되었는가?
- 왜 멈췄는가?
- 아직 사람이 확인해야 하는 것은 무엇인가?
- 계속 진행해도 안전한가?

## 장기: 팀 단위로 확장해서 사용할 수 있게 만들기

단일 Agent 데모는 유용합니다. 팀 수준에서 사용하려면 더 강한 운영성이 필요합니다.

계획:

- 여러 작업을 안전하게 동시에 실행.
- 서로 다른 프로젝트와 작업을 위해 별도 작업 공간과 기록 유지.
- 작업 유형에 따라 다른 Agent 선택.
- 계정, 자격 증명, 한도, 비용을 더 명확히 관리.
- 팀 수준 실행 환경 개선.
- 재시도와 복구 개선.
- 더 명확한 사람 승인 지점 제공.
- 서로 다른 Agent와 워크플로의 실제 효과를 비교할 수 있도록 지원.

성공 기준은 팀이 안전, 비용, 품질을 통제하면서 Agent 사용을 점진적으로 확대할 수 있는 것입니다.

## 문서와 커뮤니티

Maestro는 강력해 보이기 전에 먼저 이해하기 쉬워야 합니다.

계획:

- 메인 README를 짧고 예시 중심으로 유지.
- 깊은 기술 세부사항은 별도 문서로 이동.
- 영어와 중국어 간체 문서를 적극 유지.
- 다른 언어 번역을 제공하고 커뮤니티 개선을 환영.
- 프로젝트 시스템, Agent, 코드 플랫폼, workflow template 기여 가이드 추가.
- 더 많은 실제 엔지니어링 시나리오 예시 공개.

성공 기준은 기여자가 전체 코드베이스를 읽기 전에 유용한 시작점을 찾을 수 있는 것입니다.

## 지금은 목표가 아닌 것

Maestro는 팀의 리뷰, 테스트, 릴리스 판단을 우회하기 위한 도구가 아닙니다.

우리가 더 중요하게 보는 것:

- Agent를 실제 작업에 연결.
- 코드 출처와 작업 출처를 보이게 함.
- 실행 과정을 추적 가능하게 함.
- 고위험 단계에서 사람의 통제 유지.
- 유용한 실행 기록 보존.
- 신뢰가 쌓이는 만큼만 자동화 확대.

자동화는 희망이 아니라 증거에 맞춰 확대되어야 합니다.

---

## 현재 초점

현재 초점은 Maestro를 더 이해하기 쉽고, 더 시험하기 쉽고, 더 안전하게 평가할 수 있게 만드는 것입니다.

1. 공개 README 단순화.
2. 쉬운 언어의 roadmap 추가.
3. 로컬 데모 안내 개선.
4. 현재 연동 지원을 설명하되 외부 시스템을 “내장”이라고 부르지 않기.
5. 격리된 작업 공간의 가치를 설명.
6. TAPD, Linear, GitHub, CNB, 실제 Agent 조합 예시 추가.
7. 기술 세부사항은 제공하되 모든 새 독자가 처음부터 읽게 만들지 않기.

## 기여 방법

유용한 기여:

- 더 나은 예시.
- 더 명확한 문서.
- 더 안전한 workflow template.
- 새로운 프로젝트 시스템 연동.
- 새로운 Coding Agent 연동.
- 새로운 코드 플랫폼 연동.
- dashboard 개선.
- 실제 워크플로 테스트 커버리지.
- 원어민 번역 검수.

먼저 로컬 memory/mock 흐름으로 시작한 뒤, 점진적으로 실제 시스템으로 이동하세요.
