# Maestro Roadmap

Языки: [English](./ROADMAP.md) · [简体中文](./ROADMAP.zh-CN.md) · [Русский](./ROADMAP.ru.md) · [More](./LANGUAGES.md)

## Цель

У Maestro простая цель:

> **Сделать AI agents проще, безопаснее и надёжнее для реальных инженерных команд.**

Многие coding agents уже умеют писать код. Командам нужно больше, чем генерация кода:

- работа должна приходить из реальных систем вроде TAPD, Linear и будущих платформ;
- код должен приходить из явно настроенного Git-репозитория и ветки;
- каждый запуск должен иметь изолированную рабочую область, чтобы задачи не мешали друг другу;
- люди должны понимать, что сделал agent, что изменилось и почему был сбой;
- рискованные шаги должны оставаться проверяемыми;
- команды должны расширять использование постепенно, а не открывать все разрешения в первый день.

Эта дорожная карта организована вокруг пользовательской ценности, а не внутренних названий модулей.

---

## Ближайший этап: проще попробовать Maestro

Новый пользователь должен понять и запустить Maestro, не изучая сначала всю архитектуру.

Планируемая работа:

- более простое локальное демо;
- более понятный Quick Start;
- screenshots, GIFs или короткие видео;
- примерные задачи, показывающие полный поток;
- ясное объяснение ценности изолированных рабочих областей: параллельность, изоляция, очистка, reviewability;
- объяснение оставшихся совместимых имён `symphony`;
- понятный путь от локального демо к реальной конфигурации.

Сценарии, которые должны стать проще для демонстрации:

- TAPD-задача в GitHub Pull Request;
- Linear-задача в GitHub Pull Request;
- анализ требований до coding;
- triage входящих задач;
- предложения reviewer;
- сравнение Codex, Claude Code и OpenCode на похожих задачах.

Успех означает, что новый читатель за несколько минут может ответить:

> «Что делает Maestro и зачем это может быть нужно моей команде?»

## Далее: связать agents с реальными проектными workflow

Maestro должен помогать agents работать из систем, которыми команды уже пользуются, а не заставлять создавать новую очередь задач.

Планируемая работа:

- улучшить текущие TAPD и Linear flows;
- сделать статусы, комментарии, ссылки и результаты более понятными;
- упростить поиск, копирование и адаптацию workflow templates;
- поддержать больше типичных инженерных задач: bug fixes, small features, анализ требований, уточнение задач, triage и review suggestions;
- чётко отделить текущую поддержку интеграций от будущих целей расширения;
- подготовиться к интеграциям вроде Jira, GitHub Issues, GitLab, Gitea, Bitbucket и Feishu Project.

Успех означает, что команды могут стартовать из своего существующего проектного workflow, не меняя управление работой только ради agents.

---

## Средний этап: сделать работу agent более доверенной

Команда не должна доверять запуску только потому, что agent сказал «готово».

Планируемая работа:

- более понятная история запусков;
- более читаемые summaries;
- лучшие связи между задачами, Git-изменениями, логами и материалами review;
- более ясные причины ошибок;
- лучшая redaction логов;
- более полезный dashboard;
- видимые checkpoints перед записью в реальные проектные системы, push branches или созданием PRs;
- чёткое разделение между локальным демо, trusted evaluation, team pilot и production operation.

Успех означает, что reviewer может ответить:

- Что сделал agent?
- От какой задачи и Git-репозитория он работал?
- Что изменилось?
- Почему он остановился?
- Что ещё требует human confirmation?
- Безопасно ли продолжать?

## Долгосрочно: помочь командам использовать agents в масштабе

Демо с одним agent полезно. Командное использование требует более сильной эксплуатации.

Планируемая работа:

- безопасно запускать несколько задач одновременно;
- держать отдельные workspaces и записи для разных проектов и задач;
- выбирать разных agents для разных типов задач;
- понятнее управлять accounts, credentials, quota и cost;
- улучшать runtime environments для команд;
- улучшать retry и recovery;
- поддерживать более понятные точки human approval;
- помогать сравнивать реальную эффективность разных agents и workflows.

Успех означает, что команды могут расширять использование agents постепенно, удерживая безопасность, стоимость и качество под контролем.

## Документация и сообщество

Maestro должен быть понятным до того, как покажется мощным.

Планируемая работа:

- держать основной README коротким и основанным на примерах;
- переносить глубокие технические детали в отдельные документы;
- активно поддерживать English и Simplified Chinese;
- сохранять другие переводы и принимать улучшения от сообщества;
- добавить contribution guides для проектных систем, agents, кодовых платформ и workflow templates;
- публиковать больше примеров реальных engineering scenarios.

Успех означает, что contributors найдут полезную точку входа, не читая весь codebase сначала.

---

## Пока не цели

Maestro не пытается помогать командам обходить review, tests или release judgment.

Нам важнее:

- соединять agents с реальными задачами;
- делать источник кода и источник задачи видимыми;
- сохранять процесс трассируемым;
- оставлять human control на рискованных шагах;
- сохранять полезные записи запусков;
- расширять автоматизацию только по мере роста доверия.

Автоматизация должна расти вместе с доказательствами, а не с желаниями.

## Текущий фокус

Текущий фокус — сделать Maestro проще для понимания, запуска и безопасной оценки:

1. упростить публичный README;
2. добавить roadmap простым языком;
3. улучшить руководство по локальному демо;
4. описывать текущую поддержку интеграций, не называя внешние системы «встроенными»;
5. объяснить, почему важны изолированные workspaces;
6. добавить примеры для TAPD, Linear, GitHub, CNB и реальных agent combinations;
7. оставить технические детали доступными, но не заставлять каждого нового читателя начинать с них.

- лучшие examples;
- более понятная documentation;
- более безопасные workflow templates;
- новые project-system integrations;
- новые coding-agent integrations;
- новые code-platform integrations;
- улучшения dashboard;
- тестовое покрытие реальных workflows;
- проверка переводов носителями языка.
