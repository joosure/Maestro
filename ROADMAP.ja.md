# Maestro Roadmap

言語: [English](./ROADMAP.md) · [简体中文](./ROADMAP.zh-CN.md) · [日本語](./ROADMAP.ja.md) · [More](./LANGUAGES.md)

## 目標

Maestro の目標はシンプルです。

> **AI Agent を、実際のエンジニアリングチームでより簡単に、より安全に、より信頼できる形で使えるようにする。**

多くの Coding Agent はすでにコードを書けます。しかしチームに必要なのは、単なるコード生成ではありません。

- 作業は TAPD、Linear、将来のプロジェクト管理システムから来るべきです。
- コードは明示的に設定された Git リポジトリとブランチから来るべきです。
- 各実行には分離されたワークスペースが必要です。タスク同士が干渉しないためです。
- 人は Agent が何をしたか、何を変更したか、なぜ失敗したかを理解できるべきです。
- 高リスクのステップはレビュー可能であるべきです。
- チームは初日から全権限を開くのではなく、段階的に利用範囲を広げられるべきです。

このロードマップは、内部モジュール名ではなくユーザー価値を軸にしています。

---

## 短期：Maestro を試しやすくする

初めて触れるユーザーが、全体アーキテクチャを先に学ばなくても Maestro を理解し、実行できるようにします。

予定していること：

- より簡単なローカルデモ。
- より分かりやすい Quick Start。
- スクリーンショット、GIF、短いデモ動画。
- 全体の流れが分かるサンプルタスク。
- 分離ワークスペースの価値を明確に説明すること：並列実行、隔離、片付け、レビューしやすさ。
- 残っている `symphony` 互換名の説明。
- ローカルデモから実プロジェクト設定へ進む明確な道筋。

見せやすくしたいシナリオ：

- TAPD タスクから GitHub Pull Request へ。
- Linear タスクから GitHub Pull Request へ。
- コーディング前の要件分析。
- 新規タスクのトリアージ。
- reviewer 提案。
- Codex、Claude Code、OpenCode を類似タスクで比較する。

成功の目安は、新しい読者が数分で次を答えられることです。

> 「Maestro は何をするのか。自分のチームに必要かもしれない理由は何か。」

---

## 次：Agent を実際のプロジェクトワークフローへつなぐ

Maestro は、チームがすでに使っているプロジェクト管理システムから Agent を動かすべきです。新しいタスクキューを無理に作らせるべきではありません。

予定していること：

- 現在の TAPD と Linear の流れを改善する。
- タスク状態、コメント、リンク、結果を分かりやすくする。
- workflow template を見つけやすく、コピーしやすく、調整しやすくする。
- バグ修正、小機能、要件分析、タスク整理、トリアージ、レビュー提案など、より多くの開発タスクを扱えるようにする。
- 現在の連携サポートと将来の拡張候補を明確に分ける。
- Jira、GitHub Issues、GitLab、Gitea、Bitbucket、Feishu Project などへの拡張に備える。

成功の目安は、チームが Agent を使うためだけに作業管理のやり方を変えなくてよいことです。

---

## 中期：Agent の実行をより信頼できるものにする

Agent が「完了」と言っただけで、チームがその実行を信頼するべきではありません。

予定していること：

- より明確な実行履歴。
- 読みやすい実行サマリー。
- タスク、Git 変更、ログ、レビュー資料のつながりを改善する。
- 失敗理由を分かりやすくする。
- ログのマスキングを改善する。
- より便利な dashboard。
- 実プロジェクトシステムへの書き戻し、ブランチ push、PR 作成の前に見えるチェックポイントを置く。
- ローカルデモ、信頼できる評価、チーム試行、本番運用を明確に分ける。

成功の目安は、レビュアーが次を答えられることです。

- Agent は何をしたか。
- どのタスクと Git リポジトリから作業したか。
- 何が変更されたか。
- なぜ止まったか。
- まだ人間の確認が必要なことは何か。
- 続行して安全か。

---

## 長期：チームでスケールして使えるようにする

単一 Agent のデモは有用です。チームレベルで使うには、より強い運用性が必要です。

予定していること：

- 複数タスクを安全に同時実行する。
- プロジェクトとタスクごとにワークスペースと記録を分ける。
- タスク種別に応じて別の Agent を選ぶ。
- アカウント、認証情報、利用上限、コストを分かりやすく管理する。
- チーム向け実行環境を改善する。
- リトライと復旧を改善する。
- 人による承認ポイントを分かりやすくする。
- Agent とワークフローの実際の効果を比較できるようにする。

成功の目安は、チームが安全性、コスト、品質を管理しながら段階的に Agent 利用を広げられることです。

## ドキュメントとコミュニティ

Maestro は、強力に見える前に、まず理解しやすいべきです。

予定していること：

- メイン README を短く、例中心に保つ。
- 深い技術詳細は別ドキュメントへ移す。
- 英語と簡体字中国語を積極的に維持する。
- 他言語翻訳を提供し、コミュニティの改善を歓迎する。
- プロジェクト管理システム、Agent、コードプラットフォーム、workflow template の貢献ガイドを追加する。
- 実際のエンジニアリングシナリオ例を増やす。

成功の目安は、貢献者がコードベース全体を読む前に有用な入口を見つけられることです。

---

## 今は目指さないこと

Maestro は、チームのレビュー、テスト、リリース判断を飛ばすためのものではありません。

重視すること：

- Agent を実タスクへ接続すること。
- コードの出所とタスクの出所を見えるようにすること。
- 実行プロセスを追跡可能にすること。
- 高リスクステップでは人間の制御を残すこと。
- 有用な実行記録を残すこと。
- 信頼が積み上がった分だけ自動化を広げること。

自動化は願望ではなく、証拠に合わせて広げるべきです。

---

## 現在の重点

現在の重点は、Maestro を理解しやすく、試しやすく、安全に評価しやすくすることです。

1. 公開 README を簡潔にする。
2. 平易な言葉の roadmap を追加する。
3. ローカルデモの案内を改善する。
4. 現在の連携サポートを説明し、外部システムを「内蔵」と呼ばない。
5. 分離ワークスペースの価値を説明する。
6. TAPD、Linear、GitHub、CNB、実 Agent の組み合わせ例を追加する。
7. 技術詳細を利用可能にしつつ、新しい読者に最初から押し付けない。

---

## 貢献方法

有用な貢献には次のものがあります。

- より良い例。
- より分かりやすいドキュメント。
- より安全な workflow template。
- 新しいプロジェクト管理システム連携。
- 新しい Coding Agent 連携。
- 新しいコードプラットフォーム連携。
- dashboard 改善。
- 実ワークフローのテストカバレッジ。
- ネイティブ話者による翻訳レビュー。

まずはローカル memory/mock フローから始め、その後少しずつ実システムへ進んでください。
